# Mission Engine 資料格式規範（給內容端對齊用）

> 這份文件是給負責寫 Mission 內容（劇本/線索/對話）的人對齊資料格式用，不是工程文件。
> 對應的實際型別定義在 SS 專案 `src/app/experience/types.ts`，引擎本身不含任何 Mission 專屬邏輯——
> 新增一個 Mission，只需要照這個格式產出一份資料，不需要改任何程式碼。
>
> **這份文件只涵蓋「執行時會被引擎讀取/渲染」的正式輸出格式。** 動筆寫這份 JSON 之前，先完成
> `mission-writing-rules.md` 的每場景設計檢查（Scene Goal / Expected Question）——那些是幫助你想清楚
> 這一幕要做什麼的寫作工具，完稿後不會出現在這份 JSON 裡，就像小說的大綱不會印進成書。

## 整體結構

一個 Mission 由六個部分組成：

```
Mission
├── scenes（場景清單）
├── npcs（NPC清單）
├── clues（線索清單）
├── hypotheses（案件假說清單——玩家探索時可以隨時提出/修改的推理工具）
└── truth（真相文字，玩家結案時看到的內容）
```

**核心玩法是「玩家建立假說」，不是「玩家收集線索」。** 探索（點 Hotspot、跟 NPC 對話）只是取得證據的手段，
真正的互動核心是玩家隨時可以打開「📝 我的案件假說」工具，根據目前手上的證據，提出/修改自己相信的說法。
結案時，畫面會把玩家一路提出過的每個假說攤開來，對照真相——玩家離開時記住的不是「案件的答案」，
而是「原來我的大腦也會被這樣拼湊出來的故事騙」。這是 Mission 001 定位「認知推理（Cognitive Investigation）」
的核心機制，寫內容時要一直記得這個方向。

## 各欄位說明

### Scene（場景）

玩家會探索的一個地點。

| 欄位 | 說明 |
|------|------|
| `id` | 場景代號，英文/數字，同一個Mission內不能重複，例如 `atrium`、`info-desk` |
| `name` | 玩家看到的地點名稱，例如「中庭」「服務台」 |
| `hotspots` | 這個場景裡玩家可以點的物件清單（見下方） |

> `image`（場景背景圖）欄位**不需要內容端提供**，那是視覺呈現，等有美術素材時由工程端另外配置，內容端只要專心決定「這個場景有哪些可以探索的東西」。

### Hotspot（可探索物件）

玩家在場景裡點擊觀察/搜索的東西。

| 欄位 | 說明 |
|------|------|
| `id` | 物件代號，同Mission內不能重複 |
| `label` | 玩家看到的動作描述，例如「觀察服務台上的協尋公告」「翻找垃圾桶」 |
| `kind` | `observe`（觀察）或 `search`（搜索），純粹分類用，不影響玩法 |
| `revealText` | 玩家點下去看到的文字內容 |
| `givesClueId` | （非必填）如果這個物件會給玩家一條線索，填線索的 `id`；沒有就不填 |

> `position`（在圖片上的座標）欄位**不需要內容端提供**，等場景美術做出來後由工程端配置。

**`revealText` 要保持中性、不要下判斷。** 只描述玩家看到的事實（鞋子掉在哪、公告寫了什麼），不要幫玩家把事實
串成故事（不要寫「這代表她可能被帶走了」這種推論句）。推論應該完全由玩家自己在腦中完成，這是「玩家建立假說」
這個核心玩法能不能成立的關鍵——如果 revealText 已經幫玩家下了結論，假說工具就變成多餘的了。

### Npc（可對話角色）

| 欄位 | 說明 |
|------|------|
| `id` | 角色代號 |
| `name` | 角色名稱，例如「服務台人員」「表演工作人員」 |
| `sceneId` | 這個角色出現在哪個場景（填對應Scene的id） |
| `whyTheyThinkThis` | （非必填，會顯示給玩家）這個角色為什麼會這樣說/這樣認為，例如「只看到散場後十秒」「從二樓往下看」。用意是讓每個NPC都不是在說謊，只是每個人只看到事件的一部分，讓案件更真實 |
| `lines` | 玩家可以問的話題清單（見下方） |

**重要限制：`lines` 是固定話題清單，不是多輪對話樹。** 每個話題點下去就是一句固定的回答，不會因為玩家問了A話題就解鎖新的話題B。設計話題時，把每個NPC想成「有2-4件事想主動告訴玩家，玩家用點的方式挑順序」，不要設計成真人對話那樣一來一往深入追問。

`lines` 裡每一項：

| 欄位 | 說明 |
|------|------|
| `label` | 玩家看到的問題/話題按鈕文字，例如「詢問廣播的內容」 |
| `text` | 角色的回答內容 |
| `givesClueId` | （非必填）同Hotspot，會給線索就填線索id |

### Clue（線索）

玩家收集到的東西，會顯示在線索收藏庫裡。**這是這個產品玩法的核心，請特別注意 `firstInterpretation` 和 `finalInterpretation` 這兩個欄位一定要不一樣**，才會產生「原來如此」的翻轉感（Recontextualization）。

| 欄位 | 說明 |
|------|------|
| `id` | 線索代號，要跟某個 Hotspot 或 Npc line 的 `givesClueId` 對上 |
| `title` | 線索標題，例如「模糊的協尋公告」 |
| `firstInterpretation` | 玩家探索過程中、收集當下看到的解讀，可以是有點誤導性的第一印象（例如「玩家以為就是失蹤女孩的背包」） |
| `finalInterpretation` | 結案時看到的真實解讀，要跟 firstInterpretation 不同（例如「其實是二十歲女生的背包」） |

> 顯示時機：探索過程中，線索收藏庫顯示 `firstInterpretation`；玩家按下「我認為我知道真相了」結案時，畫面會用刪除線劃掉 `firstInterpretation`、改顯示 `finalInterpretation`，做出「你以為的 → 其實是」的翻轉效果。如果之後想要某些線索在探索「中途」就翻轉（不是全部留到結尾），要跟CC說一聲，這個顯示規則要另外設計觸發條件，目前是先全部統一在結案時翻轉。

### Hypothesis（案件假說）

玩家可以隨時打開「📝 我的案件假說」工具，從這份清單裡選一個目前最相信的說法。這個工具**不是選擇題、不是結局分支**，
玩家選幾次都可以、什麼時候選都可以，引擎只是把玩家每次選的記下來（連續選同一個不會重複記錄），結案時攤開整條歷程。

| 欄位 | 說明 |
|------|------|
| `id` | 假說代號 |
| `label` | 卡片上的文字，例如「被陌生人帶走」「自己亂跑」——用玩家會怎麼講這句話的口吻寫，不是選項式的句子 |
| `color` | （非必填）卡片顏色，十六進位色碼，例如 `#ef4444` |
| `reflection` | 玩家最後選了這個假說時，結案畫面顯示的反思文字 |

**`reflection` 的語氣非常重要，這是整個 Mission 001 定位的關鍵：不要寫成「你錯了，正確答案是___」，要寫成
「你的大腦為什麼會自然而然走到這個結論」。** 例如不要寫「陌生男子只是路人，你猜錯了」，而要寫「人看到『鞋子』
『陌生男子』『協尋』時，很容易把它們串成一個完整故事，即使三件事之間沒有證據證明彼此有關」。重點永遠是
「這是每個人都會犯的認知偏誤」，不是「你比較笨/比較粗心」。

**設計假說清單時的建議**：通常收斂成3-5個選項，包含1-2個「聽起來很合理但其實錯」的假說（呼應線索裡最強的
誤導方向）、可以留一個「尚無法判斷」讓玩家有保留判斷的空間，不強迫每個人都要有明確立場。

### truth（真相）

玩家按下「我認為我知道真相了」按鈕時看到的內容，**只有一個版本，不因玩家收集了哪些線索而不同**（沒有分支結局，沒有對/錯判定）。

| 欄位 | 說明 |
|------|------|
| `revealText` | 真相的完整說明文字 |

## 設計原則提醒（內容端寫的時候要注意）

1. **真相路徑自由，但真相本身固定**——玩家不管探索順序、探索多少，最後看到的 `truth.revealText` 都一樣，不用為不同探索路徑設計不同結局。
2. **免費內容要能獨立破關**——現階段先把探索/線索/真相這條免費主線寫完整，「付費未公開檔案」延伸內容是之後另一層，不影響這份資料格式。
3. **長度預算參考**：免費版目標15-20分鐘、完整版60分鐘，場景數與線索量可以用這個時間感抓量，不用一次設計太大。

## 格式示範（純示範用，非正式 Mission 001 內容）

```json
{
  "id": "mission-001",
  "title": "案件標題",
  "scenes": [
    {
      "id": "scene-1",
      "name": "地點名稱",
      "hotspots": [
        {
          "id": "h1",
          "label": "觀察某個東西",
          "kind": "observe",
          "revealText": "看到之後顯示的文字",
          "givesClueId": "clue-1"
        }
      ]
    }
  ],
  "npcs": [
    {
      "id": "npc-1",
      "name": "角色名稱",
      "sceneId": "scene-1",
      "whyTheyThinkThis": "這個角色為什麼會這樣說（例如：只看到十秒）",
      "lines": [
        {
          "label": "詢問某件事",
          "text": "角色的回答",
          "givesClueId": "clue-2"
        }
      ]
    }
  ],
  "clues": [
    {
      "id": "clue-1",
      "title": "線索標題",
      "firstInterpretation": "玩家收集當下以為的意思",
      "finalInterpretation": "結案時揭曉的真實意思（要跟上面不一樣）"
    },
    {
      "id": "clue-2",
      "title": "線索標題",
      "firstInterpretation": "玩家收集當下以為的意思",
      "finalInterpretation": "結案時揭曉的真實意思（要跟上面不一樣）"
    }
  ],
  "hypotheses": [
    {
      "id": "hyp-1",
      "label": "假說卡片文字（玩家口吻，不是選項式句子）",
      "color": "#ef4444",
      "reflection": "玩家選了這個假說時看到的反思文字——解釋大腦為什麼會這樣拼，不是宣判對錯"
    },
    {
      "id": "hyp-2",
      "label": "另一個假說",
      "color": "#3b82f6",
      "reflection": "同上"
    }
  ],
  "truth": {
    "revealText": "真相全文"
  }
}
```
