# Mission Writing Rules（寫作前的設計檢查，不是資料格式）

> 這份文件跟 `mission-engine-schema.md` 是兩份不同性質的文件：
> - `mission-engine-schema.md` = 最後要交給引擎執行的 JSON 格式，裡面每一個欄位都會被程式碼讀取/渲染
> - 這份文件 = 動筆寫 JSON 之前，先在自己腦中/草稿裡想清楚的設計檢查，**完稿後不會出現在輸出的 JSON 裡**，
>   就像小說寫作前的大綱不會印進成書一樣
>
> 這樣分開的原因：如果把這些寫作用的思考工具直接塞進 JSON 欄位，等於把「作者的設計筆記」寫死成「引擎的執行規格」，
> 之後想調整寫作方法（例如某些場景想同時留兩個問題）會被 schema 綁住，反而限制創作。

## 第一原則

> **玩家願意點下一個 Scene，不是因為得到答案，而是因為產生了一個非知道不可的新問題。**

這是所有 Mission 系列共通的設計哲學，寫任何一個場景前，先想這一條。我們真正要驗證的不是「玩家能不能解開案件」，
而是「玩家會不會因為新的問題而忍不住繼續探索」——這比任何獎勵機制、成就系統、強制引導都更能達成同樣的效果。

## 每個 Scene 動筆前，先完成這兩項

```
Scene Goal（場景目標）
我要改變玩家什麼認知？

Expected Question（預期問題）
我要讓玩家離開這一幕時，腦中一定帶著什麼問題？
```

寫完這兩項、想清楚這一幕的作用之後，才開始寫實際會進 JSON 的 `hotspots`、`revealText`、線索內容。

## 完成後

依照 `mission-engine-schema.md` 的格式輸出正式 JSON，這份檢查清單本身不用交出去，是幫你（作者）自己對齊用的。
